# Handoff: Mobile E2E Demo Ready

This document is for the next agent (or developer) picking up after the mobile integration session. The happy-path PDF merge flow is wired end to end. This document explains what was built, how to run a complete demo, and what is left.

## What Was Built

### API control plane (`apps/api/`)

- **`src/store.mjs`** — in-memory Maps for tasks, attachments, plans, runs, artifacts, and an idempotency set for worker events. Every function has a `// TODO: replace with Supabase` comment. Data is lost on restart; that is intentional for the demo phase.
- **`src/server.mjs`** — rewritten with a regex route table and six endpoints:
  - `POST /v1/pdf-merge/uploads` — accepts multipart PDFs, streams each to Azure Blob (`inputs` container), persists task + attachments.
  - `POST /v1/pdf-merge/plans` — loads attachments from store, calls OpenAI planner, persists plan.
  - `POST /v1/pdf-merge/approve` — persists run, enqueues worker payload, returns `{ status: "queued", taskId, runId }`.
  - `POST /internal/worker-runs/:runId/events` — authenticated, idempotent worker callback receiver. Validates `Authorization: Bearer RAINCLOUD_WORKER_CALLBACK_SECRET`, deduplicates by event id, updates task/run/artifact state.
  - `GET /v1/tasks/:taskId` — returns `{ task, plan, run, artifacts }` for mobile polling.
  - `GET /v1/artifacts/:artifactId/download` — streams merged PDF from Azure Blob.
- **`src/dispatch.mjs`** — updated to use `AZURE_STORAGE_CONNECTION_STRING` when set, falling back to `DefaultAzureCredential`. Use the connection string for local Windows development since `az` lives in WSL.

### Mobile app (`apps/mobile/`)

- **`src/api/raincloudClient.ts`** — typed API client. Base URL from `EXPO_PUBLIC_RAINCLOUD_API_URL`.
- **`src/components/TaskComposerCard.tsx`** — attach button now opens `expo-document-picker` (PDF only, max 7). Selected files appear as removable chips. `onSend` signature changed to `(text, files)`.
- **`src/screens/HomeScreen.tsx`** — wired to upload PDFs then call planner. Routes to `PlanReviewScreen` on `plan_review`.
- **`src/screens/PlanReviewScreen.tsx`** — new screen. Shows ordered file list, assumptions, steps, expected output. Reorder input re-calls planner with same `taskId`. Approve button calls `/approve` then navigates to task detail.
- **`src/screens/TaskDetailScreen.tsx`** — polls `GET /v1/tasks/:taskId` every 3 s while non-terminal. Falls back to fixture data for mock tasks. Artifact download button opens merged PDF via `Linking.openURL`.
- **`src/screens/TasksScreen.tsx`** — accepts `liveTasks` prop alongside mock tasks.
- **`App.tsx`** — wires `plan-review` route, `liveTasks` state, `onTaskUpdate` callback.

## Running the Full Demo

### Prerequisites

- Node 20+ on Windows (for `--env-file` support).
- Azure storage account with `inputs` and `outputs` containers and the `approved-worker-runs` queue.
- Azure Container Apps job `hermes-worker-job` deployed (see `infra/azure/`).
- Azure CLI in WSL, logged in with `az login`.
- Expo Go on your phone.
- ngrok (free tier is fine): https://ngrok.com/download

### Step 1 — Get the storage connection string (WSL)

```sh
az storage account show-connection-string \
  --name <storage-account> \
  --resource-group <resource-group> \
  --query connectionString -o tsv
```

### Step 2 — Set up ngrok (Windows)

```sh
ngrok http 3000
```

Copy the `https://xyz.ngrok-free.app` URL.

### Step 3 — Configure `.env` (Windows, repo root)

```sh
AZURE_STORAGE_ACCOUNT_NAME=<storage-account>
AZURE_STORAGE_CONNECTION_STRING=<output from step 1>
AZURE_INPUTS_CONTAINER_NAME=inputs
AZURE_OUTPUTS_CONTAINER_NAME=outputs
AZURE_STORAGE_QUEUE_NAME=approved-worker-runs
OPENAI_API_KEY=<your key>
RAINCLOUD_API_URL=https://xyz.ngrok-free.app   # tunnel URL from step 2
RAINCLOUD_WORKER_CALLBACK_SECRET=<any non-empty string, must match worker>
```

### Step 4 — Set the callback secret on the Azure worker (WSL)

The secret must match on both sides:

```sh
az containerapp job update \
  --name hermes-worker-job \
  --resource-group <resource-group> \
  --set-env-vars RAINCLOUD_WORKER_CALLBACK_SECRET=<same value as above>
```

### Step 5 — Start the API (Windows, repo root)

```sh
node --env-file=.env apps/api/src/server.mjs
# → Raincloud API listening on http://localhost:3000
```

### Step 6 — Configure mobile env

Create `apps/mobile/.env.local`:

```sh
EXPO_PUBLIC_RAINCLOUD_API_URL=http://<your-lan-ip>:3000
```

Find your LAN IP with `ipconfig` — use the IPv4 address under `Wi-Fi`. On iOS Simulator you can use `http://localhost:3000`.

### Step 7 — Start Expo (Windows, repo root)

```sh
npx expo start --prefix apps/mobile --host lan
```

Scan the QR code with Expo Go.

### Step 8 — Run the demo

1. Tap the attach button → pick 1–7 PDFs.
2. Type a merge prompt (e.g. `merge these, quarterly reports in order`).
3. Tap **Send** → plan review screen appears.
4. Optionally type a reorder instruction (e.g. `put Q3 before Q2`) and tap **Apply**.
5. Tap **Approve & Run** → task detail shows `queued`.
6. Trigger the worker (WSL):
   ```sh
   az containerapp job start \
     --name hermes-worker-job \
     --resource-group <resource-group>
   ```
7. Watch the task detail poll to `running` then `succeeded`.
8. Tap the download button → merged PDF opens in the system viewer.


## Known Limitations

- **In-memory store** — all task/run/artifact state is lost when the API restarts. Every demo run starts fresh. The Supabase wiring is the next step; all store functions have `// TODO: replace with Supabase` comments.
- **No API auth** — the mobile-facing endpoints have no authentication. Fine for local demo; required before any real deployment.
- **Manual worker trigger** — the worker is a one-shot Container Apps Job. It must be triggered manually via `az containerapp job start` or the Azure portal after each approval. Continuous polling or event-driven triggering is a future step.
- **Orphan blobs on error** — if an upload fails after some files are written to Azure Blob, those blobs are not cleaned up. Acceptable for demo.
- **No clarifying-question UI** — if the planner returns `needs_input`, the mobile shows an error. The API persists the state correctly; only the UI slice is missing.
- **ngrok required for E2E** — the worker callback cannot reach a plain `localhost`. The tunnel URL must be set before approving. Without it the task stays stuck on `queued`.

## Next Steps (suggested PR order)

1. **Wire Supabase** — replace `store.mjs` Map implementations with Supabase client calls. The schema is implied by the store's data shapes. Connection string is already in `.env` (`SUPABASE_URL`).
2. **Deploy the API to Azure Container Apps** — gives a stable HTTPS URL, eliminates ngrok. Update `EXPO_PUBLIC_RAINCLOUD_API_URL` in Expo build config.
3. **Add tests for new API endpoints** — especially the callback receiver (auth, idempotency, runId mismatch).
4. **Clarifying-question UI** — the `needs_input` branch is handled by the API and planner; only the mobile screen is missing.
5. **Continuous worker trigger** — event-driven via Azure Event Grid or a polling loop in the API.
